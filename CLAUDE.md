## 0. Project identity

**Name:** FaceRoots
**Tagline:** *Discover where your face comes from.*
**One-liner:** A mobile-first PWA where a user uploads portraits of themselves and their family, and the app produces a playful, narrated "Family DNA Map" showing which facial feature resembles which relative and by how much.

**This is a fun consumer app, not a forensic or genetic tool.** Every piece of copy, every LLM prompt, every share card must reinforce that framing. See В§14 for the hard rules.

**Owner:** Swastik (solo build; Lovable scaffold вҶ’ Claude Code for everything else via GitHub sync).

---

## 1. Non-negotiables вҖ” read before writing any code

1. **One Lovable prompt, ever.** Lovable has already generated the scaffold. Do not instruct the user to run any further Lovable prompts. All subsequent work ships via Claude Code вҶ’ GitHub вҶ’ Lovable sync.
2. **Privacy first.** Face data is biometric data under India's DPDP Act and GDPR. No feature ships without the consent flow wired. No raw image leaves the device until consent is recorded.
3. **pgvector, not Pinecone.** Stay inside Supabase. Do not introduce a second vector store.
4. **Edge Functions, not a FastAPI service.** All backend logic runs in Supabase Edge Functions (Deno). Heavy ML runs via hosted APIs (Replicate, fal.ai, Google AI Studio), not self-hosted.
5. **Never claim genetic relationship.** The app measures *visual resemblance*, not genetics. This phrasing is enforced in LLM system prompts, UI copy, share cards, and legal text.
6. **No ethnicity, attractiveness, health, or emotion commentary** from the LLM narration layer. Ever. Hard guardrail in every prompt.
7. **RLS on every table.** No exceptions. Every row scoped to `auth.uid()`.
8. **Model version stamping.** Every analysis row records the exact model versions used so results stay explainable across upgrades.
9. **18+ self-attestation.** v1 ships adults-only. Minors flow is a later decision with legal review.
10. **Kill switches.** Every paid API call sits behind a feature flag and a per-user rate limit.

**Principle:** landmarks, pose validation, feature cropping, **and CLIP embedding** all run fully client-side (MediaPipe WASM + Transformers.js ONNX). Crop PNGs are uploaded to Storage, embeddings are inserted directly into `feature_embeddings` by the browser. Only the cropped feature PNGs + embeddings leave the device вҖ” raw full portraits stay optional with a default 7-day TTL.

**What was removed:** the `embed-face` and `embed-features` Edge Functions have been deleted. The Replicate dependency is gone end-to-end. `face_embeddings` rows are never written.

## 4. Data model

All tables live in schema `public`. Every table has RLS enabled and a policy that scopes rows to `auth.uid()` via `owner_user_id` or via a join through `persons.owner_user_id`.

### 4.1 Core tables

### 4.1a Matching RPC вҖ” `match_feature_embeddings`

Defined as `SECURITY DEFINER` (migration 0011) so Edge Functions using the
service role can bypass RLS on `feature_embeddings`. Final shape after
migrations 0012 + 0013:

- Accepts a 512-dim query embedding.
- Restricts to **front-angle face_images only** via join to `face_images`.
- For each candidate `person_id`, picks the **single most recent** front
  face_image using `DISTINCT ON (person_id) ... ORDER BY created_at DESC`,
  so any stale duplicate from a partial cleanup cannot skew results.
- Returns `(person_id, similarity)` where similarity is the mean cosine
  similarity across that one image's feature embeddings.
- Granted to both `authenticated` and `service_role`.

### 4.2 RLS policy pattern

Every table follows this shape:

```sql
alter table <t> enable row level security;

create policy "<t>_owner_select" on <t>
  for select using (owner_user_id = auth.uid());
create policy "<t>_owner_insert" on <t>
  for insert with check (owner_user_id = auth.uid());
create policy "<t>_owner_update" on <t>
  for update using (owner_user_id = auth.uid());
create policy "<t>_owner_delete" on <t>
  for delete using (owner_user_id = auth.uid());
```

For tables without `owner_user_id` (face_images, face_embeddings, feature_embeddings, face_landmarks), policies join through `persons.owner_user_id`.

## 5. Edge Functions

All Deno, located in `supabase/functions/<name>/index.ts`. Shared code in `supabase/functions/_shared/`. Every function: validates JWT, checks rate limits, logs to Sentry, returns typed JSON.

Deployed functions (only these six exist on disk):
`validate-face`, `run-analysis`, `match-features`, `narrate-matches`,
`render-legacy-card`, `delete-my-data`. There is **no** `embed-face` or
`embed-features` function вҖ” embedding has moved to the client.

### 5.1 `validate-face`
**Input:** `{ face_image_id }` (image already uploaded by client)
**Output:** `{ valid: boolean, reason?: string, face_confidence: number, nsfw_score: number }`
**Logic:** Verifies ownership via join to `persons`, checks stored `width/height вүҘ 400` and object size вүӨ 10 MB via storage HEAD, then re-reads `face_confidence` / `nsfw_score` that the **client** wrote when MediaPipe ran. No server-side face detection, no Replicate call. Fails closed if the client didn't stamp the scores. Thresholds: `face_confidence вүҘ 0.85`, `nsfw_score вүӨ 0.3`. A browser NSFW ONNX model is a TODO.

### 5.2 `run-analysis` (orchestrator)
**Input:** `{ self_person_id }`
**Output:** `{ analysis_id }` вҖ” client subscribes via Supabase Realtime for status updates
**Logic:**
1. **Consent pre-check** вҖ” read the latest `consent_events` row for this user. Refuse if missing, if `event_type='revoked'`, or if `scopes.embeddings !== true`.
2. Verify `self_person_id` is owned by the caller and has `is_self=true`.
3. Verify the self person has вүҘ1 `face_images` row and that at least one non-self person does too.
4. Insert `analyses` row with status `pending` and respond immediately with `analysis_id`.
5. Pipeline runs via `EdgeRuntime.waitUntil(...)`. Stages:
   - `embedding` вҖ” **no server work**; verifies `feature_embeddings` already exist for self + at least one family member (the client inserted them before calling run-analysis). If self embeddings are missing, it inspects `face_images` / `face_landmarks` to report a specific reason ("Face landmarks are missingвҖҰ" vs "Feature crops or embeddings could not be generatedвҖҰ").
   - `matching` вҶ’ POSTs to `match-features`.
   - `narrating` вҶ’ POSTs to `narrate-matches`.
   - `rendering` вҶ’ POSTs to `render-legacy-card`; **failure is non-fatal** (card can be re-rendered on demand from the Share page).
   - `done` вҖ” stamps `completed_at` and `model_versions = MODEL_VERSIONS` (В§5-shared).
6. Any failure sets `status='failed'` with `error_message`.
7. Sibling Edge Functions are called with the caller's JWT (so `requireAuth` passes) using a 120 s `AbortController` timeout per call.
8. Rate limiting is scaffolded but **currently disabled** via a TODO (`checkRateLimit` from `_shared/rateLimit.ts`) вҖ” re-enable before production.

### 5.3 `match-features`
**Input:** `{ analysis_id }`
**Output:** `{ matches_written: number }`
**Logic:** Loads the self's `feature_embeddings` rows joined to `face_images` filtered to `angle='front'`. Groups by `feature_type`, computes the mean 512-dim embedding, and calls the `match_feature_embeddings` RPC (В§4.1a) with the top-N = 5. Parses pgvector values whether returned as a string `"[...]"` or as an array. For each feature: writes the top-1 as winner + up to 4 runners-up to `feature_matches` via `upsert(onConflict='analysis_id,feature_type')`. `winner_confidence = max(0, 1 вҲ’ stddev(similarities))` across the returned top-N (tight cluster вҮ’ high confidence). Logs (does not fail) when self vs family rows mix `model_version`.

### 5.4 `narrate-matches`
**Input:** `{ analysis_id }`
**Output:** `{ narrated: number }`
**Logic:** For each `feature_match` without an `llm_verdict`: fetch the two crop bytes via signed URL, hash them with SHA-256, and look up `verdict_cache` by `(user_crop_hash, winner_crop_hash, feature_type)` вҖ” cache is **content-hashed**, not path-hashed, so identical crops hit cache across re-captures. On miss, call Gemini 2.5 Flash with the locked system prompt (В§9.1). Run the regex deny-list post-filter (В§9.3); on match, fall back to the template. Write verdict back to `feature_matches` and into `verdict_cache`.

### 5.5 `render-legacy-card`
**Input:** `{ analysis_id }`
**Output:** `{ signed_url }`
**Logic:** Compose 1080Г—1920 PNG via Satori (React-to-SVG) вҶ’ resvg-js (SVGвҶ’PNG). Upload to `legacy-cards` bucket, store the object path on `analyses.card_storage_path`, and return a 15-min signed URL.

### 5.6 `delete-my-data`
**Input:** (authed, no body)
**Output:** `{ deleted: {tables: count, storage: count} }`
**Logic:** Hard-delete from all tables (cascades handle most), purge all storage objects for user, write one final `consent_events` row with `event_type='revoked'` then delete the profile. Returns receipt.

### 6.3 Front-only capture flow
The 3-angle flow was dropped in favour of a single front capture. The side
profiles weren't adding signal and they tripled the failure surface. The
RPC is hard-coded to `angle='front'` (В§4.1a), so left/right captures would
be ignored anyway.

Single screen, single step, one camera stream. State machine (see `src/pages/Capture.tsx`):

```
loading вҶ’ detecting_front вҶ’ captured_front вҶ’ uploading вҶ’ done
                                                       вҶҳ error
```

Advance rules (constants in `Capture.tsx`):
- `|yaw| < 8В° && |pitch| < 8В°` stable вүҘ 1.2 s (`STABLE_MS = 1200`)
- Face bbox **must be inside the oval overlay** (normalized oval: `cx=0.5, cy=0.5, rx=0.275, ry=0.39`)
- Face-to-frame area ratio in `[0.11, 0.32]` вҖ” neither too small nor too close

On capture: crop to an oval-aligned 768Г—1024 JPEG in canvas (the stored
portrait is already framed for thumbnail + feature cropping), haptic buzz,
shutter animation, upload.

After upload the browser immediately runs the crop + embed pipeline
(`cropAndUploadFeatures` in `src/lib/face/uploadCrops.ts`):
1. `cropFeatures` produces one PNG per `FRONT_FEATURES` entry (see В§6.2a).
2. For each crop, `embedImage` (Transformers.js CLIP ViT-B/32) produces a
   512-dim, L2-normalized `number[]`.
3. Crop PNG is uploaded to `feature-crops` at
   `{userId}/{personId}/{faceImageId}/{featureType}.png` (the user-id
   prefix is required by the bucket RLS policy).
4. A row is inserted into `feature_embeddings` with
   `model_version = CLIP_MODEL_VERSION` (`clip-vit-base-patch32-onnx-q8`).

Re-capture is handled by `replacePersonFaceImages` (`src/lib/face/replaceFaceImage.ts`) which atomically swaps the person's face_images + cascades to new embeddings, and the associated migration 0013 marks prior analyses `is_stale=true`.

### 6.3a `FRONT_FEATURES` actually used

From `src/lib/face/regions.ts`:

```ts
FRONT_FEATURES = [
  'eyes_left', 'eyes_right', 'nose', 'mouth',
  'jawline', 'forehead', 'eyebrows_left', 'eyebrows_right', 'face_shape',
]; // 9 features
SIDE_FEATURES = ['ear_left', 'ear_right']; // reserved, not currently captured
// 'hairline' is defined in FACE_REGIONS but is not in any pipeline feature list.
```

### 6.4 Family member upload flow
1. File picker (accept `image/*`).
2. Run `FaceLandmarker` in IMAGE mode.
3. If 0 faces вҶ’ friendly error with good/bad examples.
4. If >1 face вҶ’ "tap the face you want" UI (`FaceCropDialog`).
5. Auto-crop to face bbox + padding (`PhotoEditSheet` lets the user nudge).
6. Confirm: crop preview, name input, relationship dropdown.
7. Upload вҶ’ `validate-face` Edge Function вҶ’ browser runs the same
   `cropAndUploadFeatures` pipeline as В§6.3. No server embedding.

```
You are writing playful, warm, one-line captions for a fun consumer app
that compares facial features between family members.

Rules you MUST follow:
- Write exactly one sentence, max 15 words.
- Be warm, light, a little witty. Never sarcastic or mean.
- Refer to the feature by name (eyes, nose, mouth, jawline, etc).
```

### 9.2 User prompt template

```

Relationship: {relationship_tag}

[image 1: user's {feature_type} crop]
```

After the LLM responds, run a regex deny-list check for: `beautiful|ugly|pretty|handsome|attractive|dna|gene|genetic|hereditary|blood|ethnic|race|old|young|sick|healthy|fat|thin`. If any match, discard and fall back to a template like `"Your {feature} has a {similarity_pct}% echo of your {relationship_tag}."`

## 10. Validation pipeline (gibberish defense)

Layered, cheap-first:

1. **Client pre-upload (MediaPipe):** face confidence, face inside oval, bbox ratio вҲҲ [0.11, 0.32], `|yaw|<8В°`, `|pitch|<8В°`, stable вүҘ 1.2 s, single face. Scores written to `face_images.face_confidence` / `nsfw_score` before calling `validate-face`.
2. **Server post-upload (`validate-face`):** re-reads the client-written scores, re-checks ownership, bounds file size вүӨ 10 MB (HEAD), requires dims вүҘ 400Г—400. Fails closed if scores are missing. No server-side face detection today.
3. **Pose validation:** front-only capture removes the multi-angle span check. `ear_left`/`ear_right` remain in the schema for a future side-profile pass.
4. **Rate limiting:** target is 3 analyses/day free, 30/day pro. Scaffolding lives in `_shared/rateLimit.ts` but is **currently disabled** (TODO inside `run-analysis`). Re-enable before production.

## 13. Coding conventions

- **TypeScript strict mode on.** No `any`. Use `zod` for all external boundaries.
- **One default export per component file**, named matching the file.
- **No default exports in utility files** вҖ” named exports only.
- **Server-first data:** use TanStack Query, no client-side caching of Supabase rows in Zustand. Zustand is for UI state only (modal open, capture step, etc).
- **Tailwind only.** No CSS modules, no styled-components. Use `cn()` helper from shadcn.
- **Animations via Framer Motion.** No CSS keyframes unless absolutely necessary.
- **Error handling:** every Edge Function wraps its handler in `withSentry(withRateLimit(handler))`. Every fetch in the frontend uses TanStack Query error boundaries.
- **Testing:** Vitest for pure functions (cropper, pose math, guardrail filter). Playwright for critical flows (auth вҶ’ consent вҶ’ capture вҶ’ analysis вҶ’ result).
- **Commits:** Conventional Commits. `feat:`, `fix:`, `chore:`, `docs:`, `refactor:`.
- **Branches:** `main` is always deployable. Feature branches `phase-N/short-name`.

## 14. Tone & safety rules (hard)

These show up everywhere: UI copy, LLM prompts, share cards, error messages.

1. **Language:** "resemblance," "looks like," "echoes," "reminds us of." Never: "inherited," "genetic," "DNA," "bloodline."
2. **Legal disclaimer** on every results page, every share card, the consent screen, and the app footer: *"FaceRoots is a fun visual resemblance tool. It is not a genetic, paternity, or ancestry test and must not be used for any legal, medical, or identity purpose."*
3. **No attractiveness commentary.** Not in prompts, not in copy, not in achievements.
4. **No ethnicity or race commentary.** Not even positively framed.
5. **No minors.** v1 requires 18+ self-attestation at signup. Any flow for child photos is out of scope until legal review.
6. **Feedback loop:** every result card has a "This feels off" button that writes to a moderation queue for review.

## 15. Known risks and mitigations

Document live in `RISKS.md`. Initial entries:

| Risk | Likelihood | Mitigation |
|---|---|---|
| Face models have accuracy disparities across skin tones | High | Phase 6 internal audit; per-feature confidence surfacing; "this feels off" feedback |
| User uploads someone else's photo (non-consensual) | Medium | ToS warning at upload, no face-search across users, per-user RLS isolation |
| LLM outputs something mean or biased | Medium | Locked system prompt + regex post-filter + template fallback |
| Bad actors flood API via automation | Medium | Supabase Auth required, per-user rate limits, Cloudflare Turnstile on auth |
| Raw images leak via signed URL | Low | вүӨ15 min expiry, HTTPS only, TTL lifecycle, default 7-day purge |
| Model upgrade breaks old results | Low | `model_versions` stamped on every analysis; old results remain viewable |
| DPDP/GDPR compliance gap | Medium | Consent events logged, deletion endpoint, legal review before launch |
